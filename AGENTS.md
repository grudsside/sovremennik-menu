# AGENTS.md — правила работы Codex с проектом «Современник»

## Назначение

«Современник» — внутреннее веб-приложение для сотрудников кофейни. Оно объединяет методичку, обучение, техкарты, чек-листы, ревизии, задачи, расписание, ошибки, управление сотрудниками, контроль и push-уведомления.

## Репозиторий и backend

Работай только с:

```text
grudsside/sovremennik-menu
```

Актуальный backend — Supabase:

- Auth;
- PostgreSQL;
- RLS;
- Storage;
- Edge Functions;
- Web Push.

Google Apps Script не используется. Не возвращай его в проект.

## Перед изменениями

Прочитай все документы в `docs/`, фактическое дерево репозитория, историю последних commits, все SQL-файлы и Edge Functions.

## Общение

Общайся на русском и простыми словами. Владелец не обязан знать программирование. Объясняй влияние на сотрудников, Supabase, публикацию, проверку и откат.

## Когда задавать вопросы

Обязательно спрашивай, если ответ влияет на бизнес-логику, роли, RLS, структуру таблиц, миграцию, удаление данных, авторизацию, получателей уведомлений, production или интеграцию с iiko.

Codex может задавать столько вопросов, сколько действительно нужно для качественного результата. Не повторяй уже отвеченные вопросы.

## Источник истины

Приоритет:

1. фактически опубликованный Supabase;
2. ветка `main`;
3. применённые SQL-патчи;
4. документы;
5. старые override-блоки только как история.

Наличие SQL-файла не доказывает, что он выполнен.

## Supabase

Перед изменением базы:

1. перечисли таблицы, функции и политики;
2. проверь production-схему;
3. создай новый идемпотентный SQL-патч;
4. не переписывай старые применённые steps;
5. добавь проверочный запрос;
6. опиши применение и откат;
7. проверь каждую роль.

Никогда не использовать `service_role` в браузере.

## Edge Functions

Проверяй JWT, active profile, роль, входные данные, CORS, повторы и дубли. Не логируй секреты. Указывай точную команду deploy.

## Технический долг app.js

`assets/js/app.js` содержит базовую логику и override-блоки `v21–v26`. Последнее объявление функции переопределяет раннее.

Не удаляй legacy-код и не перемещай `init()` без полного тестирования. Не добавляй новый большой override-блок без необходимости. Рефакторинг выполнять отдельно от новой функции.

## Безопасность

Не коммить:

- `SUPABASE_SERVICE_ROLE_KEY`;
- `VAPID_PRIVATE_KEY`;
- `NOTIFICATION_CRON_SECRET`;
- JWT;
- пароли;
- дампы Auth;
- персональные данные.

В клиенте остаётся legacy built-in admin. Не размножай его. Удаление проводить отдельной безопасной миграцией.

## Git

Работай в отдельной ветке. Не пушь напрямую в `main`. Один PR — одна логическая задача. Не смешивай рефакторинг и новую функцию.

## Итог задачи

Укажи:

1. что изменено;
2. файлы;
3. таблицы/RLS/Functions;
4. проверки;
5. ручные действия;
6. откат;
7. оставшиеся риски.
