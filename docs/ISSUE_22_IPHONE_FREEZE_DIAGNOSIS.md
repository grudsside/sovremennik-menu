# Диагностика issue #22: зависание «Моих задач» на iPhone

## Вывод

Причина зависания — не повторная загрузка задач и не бесконечный `MutationObserver`. Проблему создаёт мобильная компоновка формы задачи в последнем CSS-слое:

- весь экран занят `position: fixed`;
- внутри него создан ещё один абсолютный scroll-контейнер;
- высота этого контейнера привязана к `100dvh`;
- прокрутка `body` одновременно заблокирована.

Когда iOS-клавиатура меняет visual viewport, WebKit должен одновременно пересчитать `dvh`, положение fixed/absolute-контейнеров, внутреннюю прокрутку и положение активного поля. На слабом или уже нагруженном iPhone эта цепочка проявляется как рывки и зависание формы.

## Доказательства

Проверка выполнена относительно `main` на commit `67b099a`.

1. Опубликованная страница загружает слои в таком порядке: `app.js`, `push-legacy.js`, `interface-redesign.js`, `interface-v3.js`, `tasks-hotfix.js`, `interface-followup.js`, `mobile-tasks-performance.js`, `mobile-active-panel.js`. Значит, итоговые правила из `mobile-tasks-performance.css` действительно управляют формой.
2. История Git показывает, что опасная комбинация появилась в commit `f8f4738`: `.task-modal` стал fixed-контейнером, а `.task-form-card` — абсолютным контейнером с `height: 100dvh` и собственной вертикальной прокруткой.
3. В коде нет обработчика `visualViewport`. Единственный обработчик обычного `resize` в финальном мобильном слое только синхронизирует CSS-класс. Автотест отправляет 50 `resize`-событий и получает нулевые счётчики для `renderApp`, `setTop`, `refreshTasks`, `loadTasks` и `loadTaskAssignees`.
4. Наблюдатели DOM запускаются только после реальных изменений DOM. Появление клавиатуры само по себе DOM не меняет. Асинхронная замена списка сотрудников происходит один раз после загрузки, поэтому она может добавить разовую задержку, но не объясняет повторное зависание при каждом открытии клавиатуры.
5. WebKit описывает те же ограничения: fixed-позиционирование становится ненадёжным при программной клавиатуре ([bug 191204](https://bugs.webkit.org/show_bug.cgi?id=191204)), `body { overflow: hidden }` ведёт себя нестабильно при уменьшенном visual viewport ([bug 240860](https://bugs.webkit.org/show_bug.cgi?id=240860)), а высота visual viewport может обновляться только в конце анимации клавиатуры ([bug 265578](https://bugs.webkit.org/show_bug.cgi?id=265578)).

## Исправление

На экранах до 920 px форма теперь остаётся внутри раздела «Мои задачи» как обычная страница:

- `position: static`;
- `height: auto`;
- без собственного вертикального scroll-контейнера;
- без `100dvh`;
- документ продолжает прокручиваться обычным способом.

На desktop форма по-прежнему переносится в `body` и открывается как модальное окно. Загрузка, сохранение, завершение и удаление задач не менялись.

## Обязательная ручная проверка перед merge

Draft PR нельзя переводить в ready и объединять, пока на физическом iPhone не пройдены сценарии:

1. открыть «Мои задачи» и форму постановки задачи;
2. по очереди открыть клавиатуру в названии, авторе и описании;
3. прокрутить форму с открытой клавиатурой;
4. выбрать исполнителя и дедлайн;
5. закрыть и снова открыть форму не менее пяти раз;
6. поставить задачу, завершить её и удалить под администратором;
7. повторить под `manager`, `staff` и `trainee` в пределах их действующих прав;
8. проверить обычный Safari и установленную PWA.
