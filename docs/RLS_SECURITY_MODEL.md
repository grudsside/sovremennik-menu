# RLS-модель Современника

Этот документ описывает целевую модель после применения:

1. `supabase/sql/STEP_10_HARDEN_RLS.sql`;
2. `supabase/sql/STEP_10B_HARDEN_RLS_VIEW_PRIVILEGES.sql`.

Оба файла входят в один релиз и применяются последовательно.

## Разделение ролей

### `admin`

- видит все задачи и может менять их статус или удалять;
- управляет сотрудниками через серверную Edge Function;
- изменяет карту доступа ролей;
- изменяет общие позиции меню, техкарты и фотографии;
- видит контрольные данные, обрабатывает ошибки и управляет расписанием;
- может вносить и корректировать поля ревизии кофе.

### `manager`

- не считается администратором базы;
- видит задачи только как создатель или исполнитель;
- может завершать задачи, где он создатель или исполнитель;
- не может менять профили, роли, техкарты, фотографии и Storage;
- видит контрольные данные, обрабатывает ошибки и управляет расписанием;
- может вносить и корректировать поля ревизии кофе.

### `barista`

- видит задачи только как создатель или исполнитель;
- может завершать только назначенные ему задачи;
- отправляет собственные чек-листы и сообщения об ошибках;
- создаёт собственную операционную ревизию кофе;
- в своей ревизии изменяет только вес бункера и количество вскрытых пачек;
- не может менять списания, продажи iiko и поле проверки.

### `waiter`

- видит задачи только как создатель или исполнитель;
- может завершать только назначенные ему задачи;
- отправляет собственные разрешённые формы;
- не может создавать или изменять ревизию кофе.

## Общие правила

- внутренние таблицы недоступны неавторизованному клиенту;
- деактивированный профиль не проходит внутренние RLS-политики;
- Push subscriptions, preferences и event history доступны клиенту только для собственного `user_id`;
- в `notification_events` активный сотрудник может обновить только `read_at` собственной строки; поля доставки и содержание остаются серверными;
- `coffee_revision_report` использует `security_invoker` и применяет RLS вызывающего пользователя;
- поля автора в чек-листах, ошибках и расписании задаются серверными триггерами;
- обновление задачи ограничено колонками `status` и `completed_at`;
- участники задачи не могут менять содержание, автора, исполнителя, VIP-признак или дедлайн;
- публичным остаётся только чтение файлов из bucket `menu-photos`.

## Ревизия кофе

Таблица содержит одну общую строку на дату.

- При первой операционной отправке сервер фиксирует реального сотрудника из активного профиля.
- Повторная отправка бариста разрешена только владельцу строки.
- Ручной ввод управляющего не заменяет автора операционной ревизии.
- Управляющий может менять контрольные поля и исправлять числовые значения.
- Официант не имеет права записи.

## Задачи

Выбран ограниченный режим для `manager`:

- глобального доступа ко всем задачам нет;
- чтение разрешено автору, исполнителю и `admin`;
- создание требует активного исполнителя;
- `admin` может завершить любую задачу;
- `manager` может завершить созданную им или назначенную ему задачу;
- `barista` и `waiter` могут завершать только назначенные им задачи;
- удаление разрешено только `admin`.

## Развёртывание

Перед применением SQL требуется резервная копия базы.

После применения проверяются:

1. вход и чтение рабочих разделов каждой ролью;
2. создание и завершение задачи;
3. удаление задачи администратором;
4. отправка чек-листа и сообщения об ошибке;
5. операционная ревизия бариста;
6. ручные поля ревизии администратора или руководителя;
7. чтение раздела «Контроль»;
8. создание события расписания руководителем;
9. сохранение фото и техкарты администратором;
10. включение, отключение и тест Push;
11. блокировка деактивированного профиля.

## Откат

Не следует возвращать прежние широкие политики вручную.

При критической ошибке:

1. остановить изменения данных через приложение;
2. восстановить схему из резервной копии;
3. либо применить отдельный проверенный reverse patch;
4. повторно выполнить smoke-тест ролей и основных операций.
