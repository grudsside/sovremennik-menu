# STEP 12 — история уведомлений сотрудников

## Что меняется

- в общей шапке появляется колокольчик со счётчиком непрочитанных уведомлений;
- сотрудник видит только собственную историю;
- первые 20 записей загружаются сразу, остальные — кнопкой «Показать ещё»;
- нажатие на запись отмечает её прочитанной и открывает связанный раздел;
- открытие панели само по себе статус не меняет;
- все адресованные сотруднику события сохраняются в истории независимо от настроек Web Push;
- новые записи приходят в открытое приложение через Supabase Realtime;
- старые записи при первом применении миграции считаются прочитанными.

## Где хранятся данные

Используется существующая таблица `public.notification_events`.

Новая колонка:

```text
read_at timestamptz null
```

`read_at is null` означает «не прочитано». Время в `read_at` означает, когда сотрудник открыл уведомление в истории.

Отдельная таблица сотрудников и отдельная авторизация не создаются. Получатель по-прежнему связан с `profiles.id` через `notification_events.user_id`.

## RLS и роли

- `admin`, `manager`, `barista` и `waiter` читают только строки со своим `user_id`;
- обновлять можно только собственные строки активного профиля;
- браузеру выдаётся обновление только колонки `read_at`;
- `title`, `body`, `user_id`, тип, источник и статус доставки остаются серверными;
- `anonymous` и деактивированный профиль доступа не получают;
- администратор не получает права читать историю другого сотрудника.

## Состав релиза

- `supabase/sql/STEP_12_NOTIFICATION_HISTORY.sql` — колонка, индекс, RLS и Realtime;
- `supabase/functions/_shared/push.ts` — запись адресованного события до проверки настроек Web Push;
- `supabase/functions/notify-event/index.ts` — дата ревизии остаётся в ключе дедупликации и больше не записывается в UUID-поле `source_id`;
- `assets/js/notifications.js` — колокольчик, история, статусы, пагинация и Realtime;
- `assets/js/app.js` — передача существующего Supabase-клиента модулю истории;
- `assets/css/styles.css` и `index.html` — шапка, панель и адаптивность;
- `tools/notification-history-check.mjs` и CI — постоянная структурная проверка функции.

Новые переменные окружения не требуются.

## Порядок развёртывания

До начала сделать резервную копию схемы и таблицы `notification_events`, зафиксировать текущие версии Functions и commit frontend.

1. Полностью выполнить в Supabase SQL Editor:

```text
supabase/sql/STEP_12_NOTIFICATION_HISTORY.sql
```

2. Убедиться, что проверочный запрос вернул:

```text
status = step_12_notification_history_done
read_at_exists = true
authenticated_can_update_read_at = true
authenticated_can_update_title = false
realtime_enabled = true
```

3. Развернуть Functions, использующие общий модуль `_shared/push.ts`:

```powershell
npx.cmd supabase functions deploy notify-event --project-ref tjibbzfdughhjenumzxo --use-api
npx.cmd supabase functions deploy push-send --project-ref tjibbzfdughhjenumzxo --use-api
npx.cmd supabase functions deploy deadline-checker --project-ref tjibbzfdughhjenumzxo --use-api --no-verify-jwt
```

Для `deadline-checker` сохранить обязательную проверку `NOTIFICATION_CRON_SECRET`. Значение секрета не выводить.

4. Только после SQL и Functions опубликовать frontend обычным PR в `main`.
5. Дождаться GitHub Pages, выполнить жёсткое обновление страницы и обновить установленную PWA.

## Обязательная проверка

### Все активные роли

Отдельно проверить `admin`, `manager`, `barista` и `waiter`:

1. колокольчик виден на компьютере и телефоне;
2. открытие панели не уменьшает счётчик;
3. отображаются заголовок, текст, дата, время и статус;
4. нажатие на запись меняет только её статус и открывает нужный раздел;
5. «Отметить все как прочитанные» обнуляет счётчик без перезагрузки;
6. после выхода, повторного входа и входа с другого устройства статус сохраняется;
7. при количестве больше 20 работает «Показать ещё»;
8. при отсутствии строк показано «У вас пока нет уведомлений»;
9. при ошибке загрузки показана кнопка повторной попытки, остальные разделы продолжают работать.

### Изоляция сотрудников

1. создать уведомление для тестового сотрудника A;
2. убедиться, что A видит запись;
3. войти сотрудником B и убедиться, что этой записи и её счётчика нет;
4. прямой `SELECT` строки A под B должен вернуть пустой результат;
5. прямой `UPDATE read_at` строки A под B не должен изменить строку;
6. прямое изменение `title` собственной строки из браузера должно быть отклонено.

### Realtime и настройки Push

1. оставить приложение получателя открытым;
2. создать новую задачу или тестовое уведомление;
3. убедиться, что счётчик и список обновились без перезагрузки;
4. отключить одну категорию в настройках Push;
5. создать адресованное событие этой категории;
6. убедиться, что всплывающий Push не пришёл, но запись появилась в истории, а в БД у неё статус доставки `preference_disabled`;
7. проверить получателя без активной Push-подписки: история должна сохраниться, а в БД должен быть статус доставки `no_subscription`.

### Недоступные состояния

- `anonymous` не читает и не обновляет `notification_events`;
- деактивированный профиль не читает историю, не меняет `read_at` и не получает Realtime;
- ошибка Realtime не блокирует остальные разделы; счётчик дополнительно сверяется при возврате на вкладку.

## Безопасный откат

1. frontend вернуть предыдущим commit через revert PR;
2. развернуть предыдущие версии `notify-event`, `push-send` и `deadline-checker`;
3. удалить таблицу из публикации Realtime, если функция отключается полностью:

```sql
alter publication supabase_realtime drop table public.notification_events;
```

4. отдельным проверенным reverse patch удалить политику `notification_events_update_read_active_own`, отозвать `UPDATE (read_at)` у `authenticated` и удалить индекс `idx_notification_events_user_unread_created`.

Колонку `read_at` и значения истории при безопасном откате не удалять: они не мешают старому frontend и сохраняют данные. Удаление колонки допустимо только после отдельного решения и резервной копии.
