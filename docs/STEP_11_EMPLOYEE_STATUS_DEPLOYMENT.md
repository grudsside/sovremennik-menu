# STEP 11 — безопасное управление статусом сотрудников

## Что меняется

- кнопка удаления заменяется на «Деактивировать» и «Активировать»;
- деактивация меняет только `profiles.is_active`;
- Supabase Auth-пользователь и исторические записи не удаляются;
- собственный текущий аккаунт администратора отключить нельзя;
- последнего активного администратора отключить нельзя;
- неактивные сотрудники видны только в административном списке;
- в список исполнителей задач попадают только активные сотрудники.

## Состав релиза

- `assets/js/employee-status.js` — интерфейс и клиентская логика;
- `assets/js/push.js` — загрузка клиентского модуля;
- `supabase/functions/admin-employees/index.ts` — серверное переключение статуса;
- `tools/employee-status-check.mjs` — структурная CI-проверка.

SQL-миграция для этого релиза не требуется.

## Порядок развёртывания

1. Объединить PR с `main` после успешного CI.
2. Обновить локальный `main`.
3. Развернуть Edge Function:

```powershell
npx.cmd supabase functions deploy admin-employees --project-ref tjibbzfdughhjenumzxo --use-api
```

4. Дождаться обновления GitHub Pages.
5. Выполнить жёсткое обновление страницы или открыть приложение в приватном окне.

## Smoke-тест

1. Войти под `admin`.
2. Открыть раздел «Сотрудники».
3. Убедиться, что есть столбец «Статус».
4. Деактивировать тестового бариста или официанта.
5. Обновить приложение под этим сотрудником и убедиться, что доступ закрыт.
6. Под `admin` снова активировать сотрудника.
7. Убедиться, что вход сотрудника восстановился.
8. Убедиться, что текущий аккаунт администратора нельзя отключить.
9. Убедиться, что деактивированный сотрудник отсутствует в списке исполнителей задач.
10. Убедиться, что после активации сотрудник снова появляется в списке исполнителей.

## Откат

При клиентской ошибке вернуть предыдущую версию frontend через revert PR.

При серверной ошибке развернуть предыдущую версию `admin-employees`. Данные сотрудников восстанавливать не требуется: релиз не удаляет Auth-пользователей и исторические записи.
