# STEP 10 — развёртывание усиленных RLS-политик

## Состав релиза

Применяются последовательно:

1. `supabase/sql/STEP_10_HARDEN_RLS.sql`;
2. `supabase/sql/STEP_10B_HARDEN_RLS_VIEW_PRIVILEGES.sql`.

Frontend-изменение в `assets/js/push.js` исправляет видимость задач для их автора. Edge Functions и Secrets в этом релизе не меняются.

## Перед применением

1. Убедиться, что PR объединён с `main` и обязательный CI успешен.
2. Обновить локальный `main`.
3. Сделать резервную копию схемы и затрагиваемых таблиц.
4. Зафиксировать текущий commit SHA и время начала работ.
5. Предупредить сотрудников о коротком окне проверки.

## Порядок

1. Открыть Supabase SQL Editor.
2. Полностью выполнить `STEP_10_HARDEN_RLS.sql`.
3. Убедиться, что транзакция завершилась и появился статус `step_10_harden_rls_done`.
4. Полностью выполнить `STEP_10B_HARDEN_RLS_VIEW_PRIVILEGES.sql`.
5. Убедиться, что появился статус `step_10b_harden_rls_view_privileges_done`.
6. Не менять политики вручную через Dashboard между двумя файлами.
7. Дождаться обновления GitHub Pages и сделать hard refresh.

## Обязательный smoke-тест

### `admin`

- вход работает;
- список сотрудников открывается;
- создание и удаление тестовой задачи работают;
- задача самому себе завершается;
- раздел «Контроль» загружается;
- ручные поля ревизии сохраняются в существующую дату;
- событие расписания создаётся и удаляется;
- фото или тестовая техкарта сохраняется;
- Push test приходит.

### `manager`

- вход работает;
- чужие задачи не отображаются;
- назначенная ему задача отображается и завершается;
- созданная им задача, назначенная другому сотруднику, отображается;
- все контрольные чек-листы, ревизии и ошибки читаются;
- событие расписания создаётся;
- раздел сотрудников недоступен;
- изменение техкарты и фото недоступно.

### `barista`

- вход работает;
- собственный чек-лист отправляется;
- операционная ревизия с весом и пачками создаётся;
- повторное изменение собственной операционной ревизии работает;
- списания, продажи iiko и поле проверки через прямой запрос недоступны;
- чужая ревизия не изменяется;
- создание события расписания отклоняется.

### `waiter`

- вход работает;
- разрешённые разделы читаются;
- собственное сообщение об ошибке отправляется;
- запись в ревизию кофе отклоняется;
- создание события расписания отклоняется.

### Деактивированный профиль

- новая загрузка приложения завершает рабочую сессию;
- прямое чтение внутренних таблиц отклоняется;
- собственные Push subscription и preferences больше не читаются и не обновляются;
- Edge Functions возвращают 403.

## Проверка данных после теста

- удалить тестовые задачи и расписание;
- удалить тестовую ревизию только тогда, когда она создавалась на отдельную тестовую дату;
- убедиться, что авторы чек-листов и ошибок соответствуют профилям;
- убедиться, что ручной ввод управляющего не заменил автора операционной ревизии;
- убедиться, что `coffee_revision_report` содержит option `security_invoker=true`;
- убедиться, что у view для `authenticated` остался только `SELECT`;
- убедиться, что политики Push не содержат admin/manager select.

## Откат

Не восстанавливать старые широкие политики отдельными командами наугад.

При критическом отказе:

1. остановить операции в приложении;
2. восстановить схему из сделанной перед релизом резервной копии либо применить заранее проверенный reverse patch;
3. вернуть frontend предыдущим commit/revert PR при клиентской ошибке;
4. повторить вход и smoke-тест всех ролей;
5. сохранить описание причины и результата отката без пользовательских данных.
